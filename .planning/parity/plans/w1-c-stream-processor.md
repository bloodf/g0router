## Architecture mapping (read before judging fidelity)
9router's processor is a JS `TransformStream` over raw provider BYTES (`stream.js:73` `new TransformStream({transform(chunk...)})`): it splits lines, parses SSE, translates, filters, frames. g0router's provider layer already performs the byte→line→JSON stage inside each provider goroutine (`internal/providers/utils/sse.go` `SSEScanner`) and hands the API layer a `chan *schemas.StreamChunk` (w0-e contract). The Go-native equivalent therefore splits the row across the two layers that already own those stages:
- **line-level parsing modes (PAR-TRANS-047)** → `SSEScanner` (`internal/providers/utils/sse.go`), which the row's gap column already names: "g0router `SSEScanner.Scan` handles `data:` lines and `[DONE]`; ignores `event:` and NDJSON".
- **chunk-level processing (PAR-TRANS-046/048/049)** → a new `internal/translation/stream.go` consuming the chunk channel and writing framed SSE. Behavior is asserted at the same observable boundary as 9router's: the bytes written to the client.
This is the same Go-native mapping standard used for w0-e (in-band `Error` field replacing JS closure error paths, GATE-RESOLUTION.md w0-e addendum).
