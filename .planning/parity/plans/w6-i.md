### 1.3 Chat-stream harness strategy (binding decision ‚ÄĒ @ai-sdk/react vs the inference mock)

**The surface.** The basic-chat page lets the operator pick a provider/model, type a
message, send it to the gateway, and watch the assistant turn STREAM back. The chat
e2e (`chat.spec.ts:14-20`) fills `input[aria-label="Message"]`, presses Enter, and
asserts the body eventually contains `"mock assistant"`.

**The mechanism that already exists (evidence).** Unlike the console SSE (server-push
`EventSource`, ¬ß1.5), the chat completion is a **one-shot POST that returns a
streaming response body** (`Content-Type: text/event-stream`). `page.route` CAN
fulfill this: the e2e `inference.ts` handler (`inference.ts:4-23`) intercepts `POST
/v1/chat/completions` and `route.fulfill`s a complete SSE body containing
`delta.content` = `Hello! I'm a mock assistant. You said: "‚Ä¶"` then `[DONE]`
(`inference.ts:9-20`). So a `fetch`/`@ai-sdk` POST to `/v1/chat/completions` under
the harness receives the full streamed assistant turn deterministically ‚ÄĒ NO fixture
EventSource needed for chat.

**Decision (binding) ‚ÄĒ four points, no fixture edit, no mock-body change needed:**
1. **The chat page POSTs to the REAL gateway route `/v1/chat/completions`** (the
   path the `inference.ts` mock fulfills and the real Go serves, ¬ß1.2). It does NOT
   POST to an admin chat-sessions endpoint for the send/receive turn.
2. **Streaming via `@ai-sdk/react`:** the page uses `@ai-sdk/react` (`useChat` /
   transport) configured to call `/v1/chat/completions` with the OpenAI chunk
   format the mock emits, OR ‚ÄĒ if the installed `@ai-sdk/react@3` transport surface
   does not cleanly map to a raw OpenAI SSE endpoint without an adapter ‚ÄĒ a thin
   **plain-fetch streaming reader** (read the `ReadableStream`, parse `data:` lines,
   append `delta.content`). The worker picks whichever maps cleanly to the
   `inference.ts` chunk shape at T3; BOTH satisfy the spec and add NO dependency
   (`@ai-sdk/react` + `ai` are installed, ¬ß1.1). The grep proof (¬ß5) asserts
   `@ai-sdk/react` is imported by the chat component (the parity intent) OR, if the
   plain-fetch fallback is taken, that the component reads `/v1/chat/completions` as
   a stream ‚ÄĒ the chosen approach is recorded at T3.
3. **The assistant turn renders into the DOM** so `chat.spec.ts` sees "mock
   assistant": the message list appends streamed `delta.content`. The input carries
   `aria-label="Message"` (the spec selector) and submits on Enter.
4. **e2e is deterministic:** the `inference.ts` mock fulfills synchronously with the
   full SSE body; no timing race. w6-i CONSUMES `inference.ts` UNCHANGED (it already
   emits a valid OpenAI stream); it corrects the mock BODY only if a real-Go-shape
   conflict surfaces at T1 (¬ß1.4) ‚ÄĒ none expected.

**Escalation trigger (¬ß8 ESCALATION-3):** if at T3 `@ai-sdk/react@3`'s transport
cannot be pointed at the raw `/v1/chat/completions` SSE without a new adapter
dependency, the page uses the plain-fetch streaming reader (point 2) ‚ÄĒ this is NOT an
escalation, it is the sanctioned in-plan fallback (no dep). ESCALATION-3 fires ONLY
if NEITHER `@ai-sdk/react` NOR plain-fetch can consume the `inference.ts` SSE shape
under the harness (it must ‚ÄĒ the mock emits standard OpenAI chunks); record the T3
decision either way.

4. **w6-i does NOT edit `ui/e2e/mocks/fixture.ts`** ‚ÄĒ the existing
